## Nettoyage — à exécuter dans Supabase SQL Editor après l'audit

Schéma réel (`src/lib/database.types.ts`, confirmé contre les migrations) :
- `app_users` n'a **pas** de colonne `email` — l'email vit uniquement dans `auth.users` (Supabase Auth), lié par `app_users.id = auth.users.id` (trigger `handle_new_user`, `supabase/migrations/0010_fix_handle_new_user_trigger.sql`). Toute identification par email doit donc passer par une jointure/sous-requête sur `auth.users`.
- `bookings` n'a pas de colonne `user_id` — la colonne réelle est `client_id` (+ `client_email`/`client_name` dénormalisés). Le template fourni utilisait `user_id`, corrigé ci-dessous.
- Aucun flag `is_test`/`is_demo` n'existe sur `app_users` ni `businesses` aujourd'hui. Identification proposée pour ce nettoyage : `auth.users.email LIKE '%@book-n-pay.invalid'` (comptes) + `businesses.name LIKE '[TEST 16/07]%'` (business). Robuste tant que la convention du jour est respectée à la lettre.
- FK cascade réelles connues avec certitude (migrations présentes dans le repo) : `staff_schedules` et `staff_absences` sont `ON DELETE CASCADE` depuis `staff`/`businesses` — pas besoin de les cibler explicitement, mais je les inclus quand même ci-dessous par prudence (no-op si déjà supprimées en cascade). Pour le reste des tables (`bookings`, `booking_members`, `chat_messages`, `booking_logs`, `services`, `business_photos`, `business_settings`, `business_reviews`, `favorites`, `referral_events`, `overage_charges`, `staff`) : la migration d'origine (`0001_schema.sql`) est absente du repo, donc le comportement CASCADE réel n'est **pas vérifiable depuis le code** — le script ci-dessous supprime explicitement dans l'ordre des dépendances (enfants avant parents), ce qui fonctionne que CASCADE soit défini ou non.

-- Nettoyage audit fonctionnel prod — 16 juillet 2026
-- Identification : email @book-n-pay.invalid (auth.users) + business

  SELECT id FROM auth.users WHERE email LIKE '%@book-n-pay.invalid'

  SELECT id FROM businesses WHERE name LIKE '[TEST 16/07]%'
),

  SELECT id FROM bookings
  WHERE biz_id IN (SELECT id FROM test_biz_ids)
     OR client_id IN (SELECT id FROM test_user_ids)
)
SELECT
  (SELECT count(*) FROM test_user_ids)    AS nb_users,
  (SELECT count(*) FROM test_biz_ids)     AS nb_business,
  (SELECT count(*) FROM test_booking_ids) AS nb_bookings;
